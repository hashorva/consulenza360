# AGENTS.md

## Project workflow

This project uses:

- GitHub as the source of truth for code and migration files.
- Supabase CLI for local database development.
- Supabase GitHub integration to deploy production migrations from the `main` branch.

## Supabase migration rules

Do not apply production database changes directly unless the user explicitly asks.

Preferred production workflow:

1. Create a migration with `supabase migration new <name>`.
2. Edit the generated SQL file in `supabase/migrations/`.
3. Commit the migration file.
4. Push to `main`.
5. Let the Supabase GitHub integration deploy the migration.

Avoid running `supabase db push` for production after GitHub integration is enabled, unless the user explicitly asks for a manual database push.

Do not modify the remote database manually from the Supabase dashboard except for emergency inspection/debugging.

Do not commit local Supabase temp files, secrets, or environment files.

## Secrets

Never commit:

- `.env`
- `.env.local`
- `.dev.vars`
- `supabase/.temp/`
- Supabase secret keys
- Cloudflare API tokens
- database passwords

## Application architecture

This is a single Cloudflare Workers Static Assets app for monitoring whether imported bond ISINs are present on Borsa Italiana EuroTLX.

Core structure:

- `src/app/`: React/Vite frontend.
- `src/index.ts`: unified Cloudflare Worker entry point for API routes, scheduled handler, and queue consumer.
- `src/worker/`: Worker-only backend modules.
- `supabase/migrations/`: database schema and RPC migrations.
- `scripts/import-isins.ts`: idempotent Excel import script.
- `wrangler.jsonc`: Cloudflare Worker, assets, cron, and queue configuration.

Frontend stack:

- React latest.
- Tailwind CSS v4.
- shadcn/ui Luma style components.
- Stone light/dark theme.
- Cyan chart accents.
- Recharts for charts.
- TanStack Table for list state.
- Hugeicons free icons.

Luma style requirements:

- Use Luma as the shadcn/ui visual baseline: rounded geometry, soft elevation, breathable spacing, and a calm product-app rhythm.
- Do not use the deprecated/default shadcn style or generic New York/default shadcn components when creating new UI.
- For new Vite setup, use shadcn/create or the shadcn CLI with the Vite template, Radix base, and the current Luma preset code generated by shadcn/create. The official Luma link currently uses preset `b2D0wqNxT`, but verify the current code in shadcn/create before running it:

```bash
npx shadcn@latest init --template vite --base radix --preset b2D0wqNxT
```

- For this existing app, inspect diffs before overwriting local components. Prefer applying the current Luma preset or reinstalling specific components intentionally:

```bash
npx shadcn@latest apply b2D0wqNxT
npx shadcn@latest add <component>
```

- Do not use `shadcn apply --only theme` when the task is to adopt the full Luma style, because Luma changes component geometry and spacing in addition to theme tokens.
- Luma changes component geometry and spacing, not only colors. When adapting existing components manually, update radius, surface treatment, shadows, gaps, and control sizing to match Luma instead of only changing theme tokens.
- Preserve this dashboard's operational density: Luma should make the UI softer and more refined, but compact ISIN rows, scan-friendly tables, and the required 44-52px row height still take priority.
- Avoid arbitrary Tailwind values such as `mb-[60px]` when a close scale utility exists. If the user gives a pixel value, choose the nearest Tailwind spacing/sizing token, for example `mb-14` or `mb-16` instead of a bracketed custom value.

## Cloudflare Worker rules

The Worker must instantiate Supabase clients only inside handler execution paths using `env.CONSULENZA360_SUPABASE_URL` and `env.CONSULENZA360_SUPABASE_SERVICE_ROLE_KEY`.

Do not create Supabase clients at module top level.

Never expose the Supabase service role key to React/Vite client code.

HTTP/API routes should validate Cloudflare Access in production using `CF-Access-JWT-Assertion`.

Scheduled and queue handlers are native Worker handlers and do not go through HTTP middleware.

Localhost may bypass Cloudflare Access only for local development when Access env vars are absent.

## Queue execution model

The daily check uses a serialized Cloudflare Queue baton-pass pattern on the Free plan.

Rules:

- Cron is `0 * * * 1-5`.
- Cron calls `maybe_start_due_run(now_utc)`.
- Supabase decides whether the configured Europe/Rome whole-hour run is due.
- When due, enqueue exactly one message: `{ run_id, iteration: 1 }`.
- Queue consumer claims up to 45 pending ISINs.
- Borsa fetch concurrency is capped at 4.
- Results are bulk-completed through one Supabase RPC.
- The consumer enqueues the next single message only if more work remains.
- Keep `max_batch_size: 1`, `max_batch_timeout: 1`, `max_concurrency: 1`, retries, and a DLQ in `wrangler.jsonc`.

Target subrequest budget per chunk is 47 external requests:

- 45 Borsa fetches.
- 1 Supabase claim RPC.
- 1 Supabase complete RPC.

## Borsa EuroTLX matching rules

v1 scope is EuroTLX only.

Endpoint:

```text
https://www.borsaitaliana.it/borsa/obbligazioni/eurotlx/ricerca-avanzata/risultati.html?isin={ISIN}&lang=it
```

Rules:

- Use `redirect: "manual"`.
- Do not use Cheerio, linkedom, jsdom, or full DOM parsing in the Worker.
- Use bounded string scanning.
- Normalize both HTML and ISIN with `.toUpperCase()`.
- Match only the exact detail-link pattern:

```text
/BORSA/OBBLIGAZIONI/EUROTLX/SCHEDA/{ISIN}-
```

- Do not use fuzzy matching or Levenshtein matching for ISIN detection.
- Do not mark a result present from a plain ISIN occurrence, because absent pages can echo the ISIN in sorting links.
- Mark the run `blocked` on repeated 403, 429, challenge-like HTML, or redirects.

## Supabase data model

The database is the source of truth after initial Excel import.

Main tables:

- `isins`: active source-of-truth instrument list.
- `check_runs`: run status, trigger type, scheduled date, counters, timestamps.
- `check_run_items`: per-run queue item state.
- `isin_checks`: historical result per `run_id + isin`.
- `settings`: enabled flag, timezone, whole-hour run time, weekday-only flag.

Required RPCs:

- `maybe_start_due_run(now_utc)`
- `claim_check_chunk(run_id, chunk_size)`
- `complete_check_chunk(run_id, results)`
- `mark_run_blocked(run_id, reason, metadata)`

Enable RLS on exposed tables. In v1, browser access should go through the Worker API only.

## UI rules

The dashboard is the first screen. Do not add a landing page.

Required UI:

- Minimal sidebar with Dashboard and Settings.
- KPI cards for active ISINs, present count, errors, and latest run.
- Cyan area chart for historical counts.
- Cyan/stone status donut for latest result split.
- Compact full-width ISIN rows, around 44-52px tall.
- Filtering, sorting, add, soft-delete, and manual run trigger.
- Live execution log visible during active runs.

Live log endpoint:

```text
GET /api/runs/:run_id/logs
```

Frontend should poll every 4 seconds while a run is active and stop when the run is completed, failed, or blocked.

Before considering backend/frontend changes complete, run:

```bash
npm run test
npm run build
npm audit --omit=dev
npx wrangler deploy --dry-run
```

Expected current state:

- Vitest parser/pool tests pass.
- Build/typecheck passes.
- Production audit has zero vulnerabilities.
- Wrangler dry-run validates Worker, assets, and queue bindings.

Wrangler may attempt to write local logs under the user home directory; sandboxed Codex sessions can block that without invalidating the dry-run if Wrangler still exits successfully.

## Deployment assumptions

- Deployment starts on `workers.dev`.
- A custom subdomain can be added later.
- Cloudflare Access protects the deployed app.
- Required production secrets must be set in Cloudflare before deploy:
  - `CONSULENZA360_SUPABASE_URL`
  - `CONSULENZA360_SUPABASE_SERVICE_ROLE_KEY`
  - `CF_ACCESS_TEAM_DOMAIN`
  - `CF_ACCESS_AUD`

If Borsa blocks Cloudflare Worker traffic, mark the run `blocked`; do not silently downgrade parser quality or add fake successful results. The next architectural decision would be switching runner or adding a real proxy integration.
